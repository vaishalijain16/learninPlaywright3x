# Source Code vs Bytecode vs Binary Code

## Example File: `01_HelloWorld.js`

```js
console.log("Hello The Testing Academy!");
```

---

## Comparison Table

| Aspect               | Source Code                                          | Bytecode                                                  | Binary Code (Machine Code)                                |
|----------------------|------------------------------------------------------|-----------------------------------------------------------|-----------------------------------------------------------|
| **What it is**       | Human-readable code written by a developer           | Intermediate, partially compiled code for a VM/runtime    | Pure machine instructions executed directly by the CPU    |
| **Readability**      | ✅ Fully readable by humans                          | ⚠️ Low-level; readable by VMs, not meant for humans       | ❌ Not human-readable — just raw 0s and 1s / hex          |
| **Portability**      | Cross-platform (runs wherever the runtime exists)    | Cross-platform (targets the VM, not a specific CPU)       | Platform-specific (tied to a CPU architecture like x86/ARM) |
| **Who generates it** | Written in an editor (VS Code, etc.)                 | Generated by the compiler/interpreter (e.g., V8 Ignition) | Generated by the compiler/JIT or assembler                |
| **Typical file ext.**| `.js`, `.py`, `.java`, `.c`                          | `.class` (Java), `.pyc` (Python), internal to V8          | `.exe`, `.o`, `.bin`                                      |
| **Execution**        | Cannot run directly — must be compiled/interpreted   | Runs inside a Virtual Machine (JVM, V8, CPython)          | Runs directly on the CPU with no intermediate layer       |
| **Speed**            | N/A (not executable)                                 | Medium-fast (VM overhead)                                 | Fastest (bare-metal execution)                            |

## How It Works for Our Example

### 1. Source Code — What You Write
```js
console.log("Hello The Testing Academy!");
```
This is plain text stored in `01_HelloWorld.js`. Anyone can read and modify it.

### 2. Bytecode — What the JavaScript Engine Creates Internally

When Node.js (powered by V8) runs this file, **V8's Ignition interpreter** compiles the source to bytecode. You don't see this intermediate form, but conceptually it looks like:

```
LdaGlobal [console]
Star r0
LdaNamedProperty r0, [log]
Star r1
LdaSMI [1]            ; argument count
LdaConstant [0]       ; "Hello The Testing Academy!"
CallProperty1 r1, r0, [0]
Return
```

This bytecode is what V8 actually executes — not the raw `.js` text, and not machine code.

> **Key point:** JavaScript is never compiled directly to machine code. It goes through bytecode first, and then the JIT (TurboFan/Maglev) may compile *hot* bytecode paths to optimized machine code at runtime.

### 3. Binary Code — What the CPU Ultimately Sees

When V8's JIT compiler decides a function is "hot" enough, it compiles the bytecode into native machine code. For an Apple Silicon (ARM64) Mac like yours, that would look like:

```
; Simplified ARM64 assembly for the same logic
adrp    x0, console_string@PAGE
add     x0, x0, console_string@PAGEOFF
bl      _GetProperty
mov     x19, x0
adrp    x0, helloworld_string@PAGE          ; "Hello The Testing Academy!"
add     x0, x0, helloworld_string@PAGEOFF
bl      _CallJSFunction
```

This is raw CPU instructions — unreadable without disassembly tools, but it runs at maximum speed with no VM overhead.

## The Compilation Pipeline (Visual)

```
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│   Source Code   │ ───► │    Bytecode     │ ───► │  Machine Code   │
│   (.js file)    │      │ (V8 Ignition)   │      │  (JIT output)   │
│                 │      │                 │      │                 │
│  Human-writable │      │  VM-executable  │      │  CPU-executable │
│  Portable       │      │  Portable       │      │  Arch-specific  │
│  Slow to parse  │      │  Fast to load   │      │  Fastest to run │
└─────────────────┘      └─────────────────┘      └─────────────────┘
```

## Quick Rule of Thumb

| If you want to...                        | You're dealing with    |
|------------------------------------------|------------------------|
| Read or edit code                        | **Source Code**        |
| Ship cross-platform (Java, Python, JS)   | **Bytecode**           |
| Ship a compiled native app (C, Rust, Go) | **Binary / Machine Code** |

---

> **TL;DR:** Source code is what you write. Bytecode is what the VM reads. Binary code is what the CPU runs. JavaScript lives in the first two — V8 handles bytecode internally, and the JIT occasionally spits out machine code for performance.
